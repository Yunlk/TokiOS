# TokiOS 设计共识

> 这不是指导书。这是备忘录。每次有人在凌晨三点问"我们当时为什么这么决定？"，答案在这里。
>
> 姊妹篇：[ARCHITECTURE.md](ARCHITECTURE.md) — 源码级架构，调用链，常见坑。

## 〇、核心哲学

**你能看到每一层。**

TokiOS 不是为了"能用"而设计的。它是为了"能看懂"。

- 内核代码必须能一次性读完（MVP 目标 ~10k 行 C）
- 没有宏魔法，没有代码生成器（除了 `gen_tk.py`，那是汇编器的活）
- 每一层的抽象都是透明的，不隐藏细节
- 遇到看不懂的地方 → 读源码 → 能懂。这就是成功

**显式优于隐式。**
- 进程就是一个 `.tk` 文件，不是 200 个内核结构的交互产物
- 权限就是 ring 层级，不是 40 种 capability
- 一切都是文件名，不是 fd 编号

**最小化 surprise。**
- 拒绝"这样更 Unix"或"这是业界惯例"作为理由
- 每个设计决策必须有具体理由，写在本文档里

## 一、TokiOS 是什么，不是什么

### 是什么

- 一个 x86 32 位保护模式教学内核
- 实验笔记的代码形态——边写边学边记
- 单人项目，所有代码一个人写
- 运行在 QEMU 上（目前），目标上真机

### 不是什么

- 不是 Linux 的替代品或竞争者
- 不是 POSIX 兼容系统
- 不是微内核
- 不是"生产可用"的操作系统
- 没有驱动框架，没有设备树，没有 ACPI

### 目标

> 写一个自己完全理解的内核。能跑自己的程序。能在上面做 galgame。

实际路径：先做一个小而完整的内核（启动→进程→文件系统→网络），然后上面能跑什么跑什么。

## 二、显式拒绝清单

这些是我们**知道**别人都做但我们**故意不做**的。

| 拒绝项 | 理由 |
|--------|------|
| `fork()` / `exec()` | 进程模型完全不同（见三） |
| 文件描述符（fd） | 一切是文件名，不是整数编号 |
| `pipe()` / 管道 | 共享内存 + hook 钩子替代 |
| POSIX 兼容 | 不需要，教学内核不是通用 OS |
| 多核 | 单核足够验证所有核心概念，SMP 是噪音 |
| 多终端 / TTY | 单终端，关注内核逻辑而非终端抽象 |
| 设备树（DTB） | 硬件枚举留给未来，现在只要键盘 |
| ACPI / UEFI | Multiboot 启动，BSP 简单够用 |
| 动态链接 / .so | `.tk` 是静态机器码，加载即跑 |
| lwIP / 第三方协议栈 | 网络栈自己写（见十），要的是理解 TCP 不是调 API |
| 微内核 IPC | 不是微内核（见下） |
| PKI（公钥签名） | 难度远超当前规模，暂退为 SHA-256 密码（见五） |
| 虚拟内存（swap） | 物理内存够用，不引入磁盘 IO 复杂度 |
| ELF 加载 | `.tk` 格式更简单（见四），ELF 分析是另一个项目的事 |
| `mmap` / `brk` | 进程用多少内存预先声明，不自增 |
| 调度优先级 / CFS | 轮转公平调度，不引入启发式 |

### 为什么不是微内核

- 微内核的 IPC 开销、消息传递、地址空间隔离是另一个维度的教学课题
- 我们想先把"一个完整的内核怎么写"搞明白
- Monolith 不代表混乱——文件分得清清楚楚，模块边界明确

## 三、进程模型

### "进程即文件"

这是 TokiOS 和所有 Unix 系最根本的分歧点。

**Unix 模型：**
```
fork()  → 复制整个地址空间 → exec()  → 覆盖为新程序
```
两个系统调用，中间有 COW（写时复制）、vfork 变体、clone 标志位……一个"启动程序"背后有几千行内核代码。

**TokiOS 模型：**
```
run myapp.tk   →  一步完成
```
`proc_load` 读取 `.tk` 文件 → 验证头部 → 映射用户页 → iret 进入 ring3。没有 fork，没有 exec，没有地址空间复制。

**这个决策的含义：**

- **没有"父子进程"概念**。没有 `getpid()` / `getppid()`。进程不是树。
- **删除 `.tk` 文件 = 杀死进程**。
  - 没有僵尸进程（不需要 `wait()` 回收）
  - 没有孤儿进程（父进程不是概念）
  - 没有匿名进程（每个进程都对应一个可见的 `.tk` 文件）
- **"多任务" = 多个 `.tk` 同时运行**（调度未实现，当前单进程）
- **进程间通信不走 fd / pipe**，走共享内存 + hook（见九）

### 为什么拒绝 fork+exec

1. **fork 是历史包袱**。1970 年代的 PDP-11 内存小，fork 匹配当时的硬件约束。今天的 CPU 有分页、有 MMU，fork 的 COW 实现是纯工程债。
2. **教学价值低**。fork 的复杂性（COW、vfork、clone flags）分散了"进程是什么"的核心理解。
3. **一步启动更透明**。你能看到：文件→内存→验证→跳转。中间没有魔法。
4. **"进程即文件"是 Plan 9 精神后裔**，但比 Plan 9 更激进——Plan 9 的 fork 还在，我们连 fork 都不要。

## 四、.tk 文件格式

```
┌─────────────────────────────────────┐
│  magic    (4B)  0x696B6F74  "toki" │  ← 小端序
│  entry    (4B)  入口相对偏移        │  ← 当前固定 0x00200000
│  code_size(4B)  纯机器码大小        │  ← 1 Byte ~ 1 MB
│  flags    (4B)  权限标志（当前 0）  │
├─────────────────────────────────────┤
│  机器码 (code_size 字节)             │  ← x86 裸指令，无重定位
└─────────────────────────────────────┘
```

### 为什么是固定 16 字节头

- 简单到可以用眼睛验证。`xxd myapp.tk | head -1` 看懂一切。
- ELF 头部 52+ 字节，含节表、符号表、重定位——我们的 `.tk` 不需要这些。
- 没有重定位：代码到 0x200000 就编译到 0x200000。

### 为什么 entry 写死 0x00200000

- 当前所有 `.tk` 程序加载到同一个物理地址。
- entry 字段预留了灵活性——未来如果有多个进程，可以映射到不同虚拟地址。
- 目前用户代码在 0x200000 - 0x2FFFFF（1MB），足够。

### 为什么没有段（section）概念

- `.tk` = 代码 + 数据一体，flat binary 风格。
- 不需要
- `.text` / `.data` / `.bss` 分离——`gen_tk.py` 生成的代码直接把数据嵌入指令流。
- 未来有需要再加，但默认简单。

### tk_validate 的三道检查

```c
magic     != "toki"  →  return -1   // 不是我们的文件
code_size == 0       →  return -2   // 空文件无意义
code_size > 1MB      →  return -3   // 拒绝巨型程序
flags     != 0       →  return -4   // 保留位，必须为零
```

## 五、权限模型

### 概述

| Ring | 谁 | 怎么进入 | 能做什么 |
|------|-----|---------|---------|
| 0 | 内核 | 始终在此 | 一切 |
| 3 | 所有用户 & `.tk` 进程 | `proc_start(iret)` | 仅 USER 页 + 系统调用 |

**默认全在 ring3。** 没有"root 用户是 ring0"这种设计——内核就是内核，用户就是用户，泾渭分明。

### 认证机制

**密码认证（SHA-256）：**
```
用户输入密码 → SHA-256 哈希 → 比对 /etc/passwd 中的 hex 哈希
```

**为什么是密码，不是密钥对？**

原计划用 Ed25519 公私钥签名验证，遇到实际问题：
- Ed25519 验证器实现 ~200 行 C，涉及大数运算，远超当前内核规模
- 公钥存储格式、签名传递协议都是额外的设计空间
- SHA-256 实现 ~150 行，已验证正确，够用

密钥对方案留到网络栈阶段——届时 Ed25519 的复杂度可以被"远程认证"这个需求支撑。

**为什么 SHA-256 hex 字符串，不是二进制？**
- `/etc/passwd` 人类可读：`tokisora:5e8848...` 一眼就能看懂
- 调试友好：vim 打开编辑，肉眼对比哈希
- 存储开销可忽略（64 字符 vs 32 字节，总共 8 用户 × 64 = 512 字节）

### 防暴力

**3 次输错 → 30 秒 busy-wait 冻结 → 计数器归零。**

为什么是 3 次不是 5 次？
- 3 次 = 自己打错 2 次的空间 + 1 次容错
- 真正忘记密码时，3 次和 5 次没区别——都要重装
- 30 秒够烦人，不会误触发但成功阻止暴力

为什么用 busy-wait 不是定时器？
- 当前没有定时器中断
- busy-wait 简单粗暴有效
- 30 秒不响应是 feature 不是 bug

### TSS IOPB

**所有 IO 端口对 ring3 设为 1（禁止）。**

没有"部分端口开放"的策略。原因：
- ring3 不需要直接操作硬件——有系统调用
- 之前 ring3 能用 `inb 0x60` 读键盘绕开 shell，这是安全漏洞
- 全禁 = 零信任 = 简单可靠

### 用户表（/etc/passwd）

**格式：** `username:sha256hex\n`，每行一个用户，最多 8 个。

**设计决策：**
- 为什么上限 8 用户？— 教学内核，不是企业级。8 个够所有测试场景。
- 为什么用户名最长 31 字符？— SHA-256 hex 是 64 字符，`MAX_USERNAME=31` + `:` + `64` + `\n` = 97 字符，一行 < 128 字节。
- 为什么不加密 `/etc/passwd`？— 哈希已经是加密的，密码的明文不在磁盘上。用户名公开即可。
- 默认账户 `tokisora`（密码同用户名）— 首次启动自动创建，避免"密码是什么"的尴尬。

## 六、系统调用设计

### 当前：int 0x80 trap gate

**为什么用 trap gate (0xEF) 而不是 interrupt gate (0x8E)？**

- Interrupt gate → CPU 自动 `cli` → 禁止嵌套中断
- Trap gate → 不自动关中断 → 允许更高优先级中断打断系统调用
- 当前单核、无非抢占，两者行为一样。trap gate 为未来留空间。

**为什么用寄存器传参而不是栈？**
- x86 的 `int 0x80` 切换到 ring0 时，CPU 自动切换栈（TSS.ss0:esp0）
- 如果 ring3 栈上有参数，ring0 handler 要解析 ring3 的 esp → 复杂
- 寄存器传参：直接读 `struct regs` 的 ebx/ecx/edx → 简单

### 调用号边界检查

```c
if (r->eax < 1 || r->eax > SYS_MAX) return;
```

无效调用号静默返回，不崩溃。原因是 ring3 程序写 bug 不应该把内核打崩。

### 未来：函数钩子跳转表

**规划取代 `int 0x80`，原因：**

| 方式 | 开销 | 特权切换 | 复杂性 |
|------|------|---------|--------|
| `int 0x80` trap gate | ~100 周期 | 有（ring3→0→3） | 低 |
| `sysenter`/`sysexit` | ~50 周期 | 有 | 中（需要 MSR 设置） |
| 函数钩子跳转表 | ~3 周期 | 无（同 ring） | 中（需要编译期符号绑定） |

钩子跳转表的思路：
- `.tk` 文件声明依赖的钩子名（如 `cout`, `getch`）
- 加载时内核把钩子名替换为实际函数地址
- 调用就是 `call [hook_table + offset]`，约 3 周期

**为什么不是现在做？**
- `int 0x80` 已经能用，先做功能再优化
- 钩子表需要 `.tk` 格式扩展（钩子名表），会影响 `gen_tk.py`
- 性能不是当前瓶颈——ring3 shell 打字的延迟感受不到 100 周期

### 调用号分配原则

- `SYS_COUT=1` 而不是 `=0`，因为 `eax=0` 是"无操作"的哨兵值
- 按功能块分组：1-4 终端 IO，5-8 文件系统，9 进程控制，10 调试，11 输入，12-13 用户管理
- `SYS_EXEC` 和 `SYS_EXIT` 不返回，语义不同，分两个调用号

## 七、文件系统设计（TokiFS）

### 为什么是自写而不是移植 FAT/ext2

- 移植别人的文件系统 = 学别人的 API
- 自己写 = 理解文件系统的本质：名字→数据 的映射
- TokiFS 的 16 槽 × 28 字符 × 1MB = 所有边界条件一目了然

### 为什么上限 16 个文件

- 够用。当前实际只有 2-3 个文件（`shell.tk`, `/etc/passwd`, user programs）
- 16 是 2 的幂，bitmask 操作舒服
- 太少（4/8）限制发展，太多（64/256）浪费内存且不必要

### 为什么文件名最长 28 字符

- 容纳 `/etc/passwd`（12 字符）+ 普通文件名（如 `my_long_app_name.tk`）
- 每个 `file_t` 正好 28 + 4 + 4 + 1 = 37 字节（实际结构体会对齐到 40）
- 16 × 40 = 640 字节的元数据，远小于 1MB 数据区

### 删除 = 紧凑回收

```
删前: [A:100B][gap][C:100B]
删后: [A:100B][C:100B]
```

**为什么不做"标记删除 + 定期整理"？**

- 16 个文件的碎片根本不是问题
- 紧凑回收的实现（数据前移）是绝佳的教学时刻——让你实际"看到"文件系统的空间管理
- 未来如果文件数量变大再做标记删除，现在简单优先

### 没有目录树

**TokiFS 是平的。** 没有 `/usr/bin/`，没有嵌套目录。

**为什么？**
- 16 个文件不需要目录——`ls` 直接看清所有文件
- 目录树增加的概念（inode、目录项、`.` 和 `..`）远超收益
- 我们明确拒绝了 `.` 和 `..` 路径——它们是 Unix 的历史债

如果未来真的需要层次结构，再加。但 16 个文件的系统不需要——这是工程判断，不是偷懒。

## 八、内存与分页

### 布局决策

```
0x100000  页目录  ← 刚好在 1MB 以上，跳过 BIOS/IVT
0x101000  页表    ← 紧密排列
0x200000  用户空间 ← 2MB 边界，容易记
0x300000  TokiFS   ← 3MB 边界，和用户空间隔 1MB
```

**为什么都是对齐到 MB 边界？**
- 人类可记。0x200000 是 2MB，0x300000 是 3MB。
- 页目录用 `vaddr >> 22`（除以 4MB）就够了——2MB 在页目录的第 0 项（因为只有 4MB 映射），不增加复杂度。

### 为什么只有 4MB 分页

- 内核本身 < 1MB，用户空间 < 1MB，TokiFS < 1MB
- 总共 < 4MB，一个页表（1024 项）就够了
- 如果未来需要更多内存，加第二个页表即可——不是架构问题，是需求问题

### Ring3 栈守护页

**在用户栈下面放一个 unmapped 页。**

```
0x2FF000  用户栈页 (PAGE_USER)    ← ESP 初始 = 0x2FFFFC
0x2FE000  守护页 (PTE=0)          ← 溢出到这里 → #PF
0x2FD000  普通页 (ring0 only)
```

**为什么是一个页不是多个页？**
- 一个 4KB 守护页能捕获绝大多数栈溢出
- 多个页是过度设计——如果栈真的溢出超过 4KB，程序本身就值得怀疑
- `page_guard_set` 是通用函数，多加页只需多调一次

## 九、共享内存（已规划，未实现）

### 设计思路

```
进程 A: tk_share("mybuf", 0x200000, 4096)
进程 B: tk_share("mybuf", 0x200000, 4096)
  → 两个进程的 0x200000 指向同一物理帧
```

### 继承规则："只往上走"

```
父进程 P（ring 3）创建共享区 S
子进程 C（ring 3）继承 S
子进程 C 崩溃 → P 继续持有 S
父进程 P 崩溃 → S 回收
```

**为什么不往下走？**
- 防止隐式提权：子进程不给父进程共享任何东西
- 父进程是创建者，拥有者，唯一能销毁共享区的人
- 简单，透明

### 为什么不用 pipe/消息队列

- Pipe 是 Unix 发明来解决"没有共享内存"时的通信问题
- 我们有分页——共享内存是天生的 IPC 机制
- 管道有缓冲区限制、阻塞语义、EOF 处理——共享内存只有一个问题要解决：同步
- 同步用原子标志位解决（不需要信号量内核对象）

## 十、网络栈（已规划，未实现）

### 为什么自写而不是移植 lwIP

- lwIP 是"调 API"——学会了用 lwIP，没学会 TCP
- 自写 ARP + IP + TCP 是"你真正懂了"的唯一证明
- TCP 状态机 ~300 行 C，没想象中复杂
- 教学内核，教学 > 生产效率

### 协议范围

- ARP：查 MAC 地址
- IP：包路由（单网卡，单子网——路由表可以是最简形式）
- TCP：可靠流（三次握手 + 滑动窗口 + 重传）
- 不做 UDP（TCP 够用）
- 不做 ICMP（ping 不必要）
- 不做 DHCP（IP 配置写死或从 Multiboot 信息获取）

## 十一、崩溃哲学

### "进程死 ≠ 系统死"

```
进程崩溃:
  → 捕获异常 (#PF, #GP, #UD 等)
  → dump 寄存器到屏幕
  → kill 进程（回收 .tk 占用的内存）
  → 可尝试 reboot 进程（重新 proc_load 同一个 .tk）
```

### #DF（双重异常）特殊处理

`exc8` 有独立 handler——不碰栈，直接用寄存器写 VGA：
```asm
exc8:
    cli
    movl $0xB8000, %edi
    movl $0x0F200F44, (%edi)     # " D"
    movl $0x0F210F46, 4(%edi)    # "F!"
    movl $0x0F210F21, 8(%edi)    # "!!"
1:  hlt
    jmp 1b
```

**原因：** 如果崩溃原因是栈溢出，`pusha`（原 common stub 做的第一件事）会再次触发异常 → triple fault → CPU 复位，什么都看不到。无栈 handler 至少能在死之前告诉你发生了什么。

### 没有内核 panic

内核 panic 是 Linux 的概念："系统遇到不可恢复的错误，停掉一切"。TokiOS 不区分"进程 panic"和"内核 panic"——当前的异常处理器对所有 ring0 异常都是 dump + halt。未来 ring0 异常可以尝试更优雅的恢复，但目前的教学目的下，dump 信息比试图恢复更有价值。

## 十二、视觉风格

### VGA 终端视效

| 要素 | 值 | 来源 |
|------|-----|------|
| 背景 | 蓝 (0x0F / 0x1F) | 复古 BIOS 蓝 |
| 高亮 | 黄 (0x0E) | 用户在 TokiOS 内部设计的 |
| 光标 | 白 (0x0F) 下划线 | VGA 默认 |
| 字体 | VGA 文本模式 8×16 | 硬件固定 |

### 为什么是蓝底

- 致敬 90 年代 BIOS setup 界面
- 和"我们能看懂每一层"的透明哲学一致——你看到的就是一块 VGA buffer，没经过任何渲染管线
- 不像现代终端（黑底绿字 hacker 风），蓝底有自己的辨识度

### README 风格

- **ASCII art logo**：而不是 PNG 图片
- **实验笔记口吻**："接了 QEMU 11 这个祖宗，Multiboot1 `-kernel` 被废了"
- **MIT 许可，写"拿去拆，别署名"**：不给使用者心理负担

## 十三、语言与工具

### C 语言（当前）

- 纯 C，不用 C++
- `-ffreestanding -nostdlib`：没有 libc 依赖
- `-fno-pie`：不需要位置无关代码
- `-mno-sse -mno-mmx`：关闭 SIMD，避免 ring3 #UD
- 内联汇编用 AT&T 语法（和 boot.S 统一）

### 计划：TkAsm / TkLang

**长远规划（MVP 之后考虑）：**

- 当前 ring3 程序用汇编写（通过 `gen_tk.py` 生成机器码）
- 如果要在 TokiOS 上做 galgame，汇编太痛苦
- 方案 A：移植一个 C 子集编译器（如 tinycc / chibicc）
- 方案 B：自写一门最小语言——TkAsm 的"高级版"，编译到 `.tk`

**决策推迟理由：** 先让内核完整。语言是"上面"的事，不是"下面"的事。

## 十四、版本与发布

### 版本号

**没有版本号。** TokiOS 不是产品，是项目。用 git commit hash 追踪状态。

### 许可

MIT。不署名的 MIT。原因：署名 = 使用者必须在你面前低头。不署名 = 拿代码去用就行，别想太多。

### 远程仓库

- 主仓库：`github.com/Yunlk/TokiOS`
- README 直连仓库里的文件
- 不维护 release / tag / changelog——看 git log 就是 changelog

## 附录 A：与类似项目的差异

| 项目 | 相似点 | 关键差异 |
|------|--------|---------|
| **xv6 (MIT)** | 同是教学内核，同用 x86 | xv6 教 "怎么做小型 Unix"（fork+exec+fd+pipe）；TokiOS 教 "不学 Unix 怎么设计 OS" |
| **Exokernel (MIT, 1995)** | 内核只做安全多路复用，应用直接操作硬件 | TokiOS 更"厚"一些——有文件系统、有 shell，不是纯 hypervisor |
| **Plan 9 (Bell Labs)** | "一切皆文件"精神祖先 | Plan 9 有 fork、有 namespace、有 9P 协议；TokiOS 更激进地拒绝 Unix 传统 |
| **TempleOS** | 单人 x86 内核，ring 0 全开 | TempleOS 全 ring0；TokiOS 有分层、进程隔离、密码认证 |
| **SerenityOS** | 单人项目，复古风格 | SerenityOS 是完整桌面 OS + 浏览器；TokiOS 是内核，不做 GUI |
